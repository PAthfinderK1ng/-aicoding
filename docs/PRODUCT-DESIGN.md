# 产品设计说明

## 项目名称

求职申请管理看板

## 设计背景

大学生在秋招、春招和实习申请阶段，通常需要同时管理多个岗位。不同公司会有不同的截止日期、材料要求和面试流程，信息往往分散在官网、招聘平台、邮件和聊天工具中，导致用户容易遗漏关键节点。

## 核心痛点

- 截止日期多且分散，容易错过投递或补材料时间
- 申请状态混乱，不知道每个岗位进行到哪一步
- 简历、作品集、成绩单等材料版本难管理
- 面试安排与 follow-up 节奏容易忘记
- 岗位描述下线后，后续准备面试时找不到原始 JD

## 设计目标

- 让用户一眼看清整体求职进度
- 把“下一步该做什么”展示得更明确
- 帮助用户保留岗位描述、材料版本和联系人信息
- 降低遗漏风险，提升求职过程中的秩序感和掌控感

## 产品结构

### 1. 首页总览

- 申请总数
- 本周将截止
- 待补材料
- 待跟进
- 已拿 Offer

### 2. 行动中心

- 今日优先处理
- 待跟进列表
- 材料与版本

### 3. 求职进度看板

阶段包含：

- 待申请
- 已投递
- 笔试中
- 面试中
- Offer
- 已结束

### 4. 近期时间线

- 截止投递时间
- 面试时间
- 跟进提醒时间

### 5. 岗位详情面板

- 岗位基础信息
- 投递渠道
- 联系人
- 简历版本
- JD 快照
- 材料状态
- 阶段行动建议
- 流程记录
- 备注

## 设计亮点

### 1. JD 快照

参考成熟求职管理工具的做法，加入岗位描述留档能力，避免岗位下线后无法回看要求。

### 2. 材料版本管理

不仅记录材料是否提交，还记录简历和作品集使用的版本，更符合真实求职使用习惯。

### 3. Follow-up 管理

把联系人、跟进日期和下一步动作放在同一套流程中，避免“投了就忘”。

### 4. 阶段行动建议

不是只显示一个状态标签，而是根据当前阶段给出可执行建议，帮助用户推进流程。

## 技术实现思路

- 前端：原生 HTML、CSS、JavaScript
- 后端：Node.js 原生 `http` 模块
- 数据存储：本地 JSON 文件

## 为什么这样实现

本题的重点是产品结构和功能设计，因此采用轻量实现方式，避免技术栈过重，同时保留真实交互、真实 API 和本地持久化，兼顾展示效果与工程完整度。
