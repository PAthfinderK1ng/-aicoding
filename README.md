# 求职申请管理看板

一个面向大学生求职场景的轻量全栈作品，用来管理多岗位并行申请过程中的截止时间、材料状态、流程推进、笔试测评和后续跟进。

这个项目不是停留在概念稿或静态原型，而是做成了可运行、可交互、可持久化的演示版本。用户可以在页面中查看岗位总览、拖拽推进状态、新增岗位、打开详情抽屉、查看 JD 与测评链接，并把变更写入本地数据文件。

## 项目背景

大学生在秋招、春招和实习申请阶段，通常会同时申请多个岗位。不同岗位的投递入口、截止日期、简历版本、材料要求和面试流程各不相同，信息又分散在官网、招聘平台、邮件和聊天工具里，导致用户很容易遇到这些问题：

- 申请岗位一多，就记不清哪些已经投递、哪些还没处理
- 截止日期和 follow-up 节点容易遗漏
- 同一个岗位到底投了哪一版简历，后面常常回忆不起来
- 岗位链接失效后，面试前无法快速找回原始 JD
- 笔试和人才测评链接分散，容易错过处理时间

这个项目的目标，就是把这些零散信息收拢到一个统一的求职管理界面里，让用户始终清楚三件事：

1. 我投了什么
2. 当前进行到哪里
3. 下一步该做什么

## 设计目标

- 提供清晰的求职流程视图，而不是简单的表格堆叠
- 让关键动作优先暴露，例如“今天该处理什么”“哪些岗位需要跟进”
- 把岗位信息、JD、简历版本、测评入口、联系人等真实投递要素整合到一起
- 在保证展示效果的同时，保留真实的交互和本地持久化能力

## 核心功能设计

### 1. 分栏式浏览结构

为了避免首屏信息过载，页面被拆成三个主要分栏：

- `总览`
  - 展示求职总量、待补材料、待跟进、Offer 等核心指标
  - 提供今日优先处理、待跟进列表、材料与版本三个行动中心模块
- `申请看板`
  - 作为核心操作区，集中展示各阶段岗位卡片
  - 支持筛选、搜索和拖拽更新状态
- `时间线`
  - 展示近期投递截止、笔试测评提醒、面试安排和跟进节点
  - 点击时间节点可直接打开对应岗位详情

这种结构相比“所有内容同屏展示”的方式更容易扫描，也更接近真实产品里的信息分层。

### 2. 看板式流程管理

岗位按以下阶段组织：

- `待申请`
- `已投递`
- `笔试中`
- `面试中`
- `Offer`
- `已结束`

每张岗位卡片都展示简明但关键的信息：

- 公司与岗位名称
- 截止日期
- 当前优先级
- 材料完成度
- 当前简历版本
- 是否已经保存 JD
- 是否已安排跟进
- 下一步动作

用户可以直接拖拽卡片改变阶段，让“状态记录”变成“流程推进”。

### 3. 详情抽屉

详情不再占据独立分栏，而是改成右侧抽屉，减少主页面信息压力。

点击岗位卡片或时间线节点后，会从右侧滑出详情抽屉，展示完整的岗位信息，包括：

- 截止日期与时间判断
- 投递渠道与来源
- 投递简历版本
- JD 原始链接
- 联系人与跟进提醒
- 岗位描述快照
- 笔试 / 人才测评安排、提醒与链接
- 材料状态与材料版本
- 阶段行动建议
- 面试 / 流程记录
- 备注与最近更新时间

这个抽屉是整套作品里最接近真实求职管理场景的部分，因为它把“看板上的概要信息”延伸成了“可复盘、可跟进的完整岗位档案”。

### 4. JD 快照与简历版本绑定

这是本项目重点补强的能力。

很多求职管理工具只会记录“公司名 + 岗位名 + 状态”，但真实使用中更常见的问题其实是：

- 原职位页面下线后，面试前找不到岗位要求
- 记不清当时投递的是哪一版简历
- 不同岗位用的是不同作品集或材料版本

因此项目里加入了：

- `JD 链接`
- `岗位描述快照`
- `简历版本`
- `材料版本`

这使得这个看板不仅能“记录进度”，还能“帮助回忆申请上下文”。

### 5. 笔试 / 人才测评提醒与链接

这一部分是针对大学生求职真实痛点做的增强设计。

项目支持在岗位信息中记录：

- 测评类型，例如笔试、人才测评
- 测评名称
- 测评提醒时间
- 测评处理时间
- 测评链接

同时这些信息会出现在：

- 详情抽屉中
- 时间线中
- 优先级判断逻辑中

这样用户不需要再去邮件或平台消息里翻测评入口。

### 6. 行动中心

为了避免用户只看到“状态”却不知道“怎么推进”，页面单独设计了行动中心：

- `今日优先处理`
- `待跟进列表`
- `材料与版本`

行动中心会结合截止时间、材料缺失、测评提醒、跟进时间等因素给出更接近执行层的提示。

### 7. 搜索与筛选

在申请看板里支持以下筛选维度：

- 关键词搜索
- 类型筛选
- 优先级筛选
- 材料状态筛选
- 阶段筛选

这样在岗位数量增加后，用户仍然可以快速定位重点岗位。

## 产品亮点总结

相较于普通的表格或简单双列看板，这个项目强化了几个更贴近实际求职过程的设计点：

- `分栏式浏览`：降低同屏信息堆叠带来的阅读压力
- `详情抽屉`：主流程轻，深信息按需展开
- `JD 留档`：避免岗位下线后丢失上下文
- `简历版本绑定`：减少“同一份简历投所有岗位”的混乱
- `测评提醒 + 链接`：把笔试和人才测评纳入统一管理
- `行动中心`：从展示状态升级到推动行动

## 技术实现

项目采用轻量全栈实现，重点是保证结构清晰、容易展示、容易运行。

### 前端

- 原生 `HTML`
- 原生 `CSS`
- 原生 `JavaScript`

前端负责：

- 数据渲染
- 分栏切换
- 看板拖拽
- 抽屉展示
- 表单录入
- 与后端 API 通信

### 后端

- Node.js 原生 `http` 模块

后端负责：

- 静态资源托管
- 获取岗位列表
- 新增岗位
- 更新岗位字段
- 持久化数据到本地 JSON 文件

## API 设计

当前项目提供以下接口：

- `GET /api/applications`
  - 获取全部岗位数据
- `POST /api/applications`
  - 新增岗位
- `PATCH /api/applications/:id`
  - 更新已有岗位字段

## 如何运行

### 推荐方式

如果环境支持 `npm`：

```bash
npm start
```

然后打开：

```text
http://localhost:3000
```

### 无 npm 的方式

如果本地只有 Node 环境，也可以直接运行：

```bash
node server.js
```

### 重置演示数据

如果你在演示过程中新增或修改了数据，想恢复到初始状态：

```bash
node scripts/reset-data.js
```

如果支持 `npm`，也可以：

```bash
npm run reset-data
```
